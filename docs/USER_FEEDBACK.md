# User Feedback / 用户反馈记录

## V2 试用反馈

### 反馈来源

本轮反馈来自 V2 线上版本的真实试用体验。

当前版本为前端 Mock 数据版本，主要用于验证手机浏览器访问、问答流程、地点选择、行程生成、地图展示区域和整体页面体验。

本轮反馈主要集中在：推荐数据真实感、攻略文本提取效果、地点展示方式、天气因素、预算表达方式等方面。

---

## 用户反馈整理

### 1. 攻略文本提取效果还不够好

反馈内容：

用户反馈“文字提取好像不是太好”，说明当前从用户粘贴的攻略文本中提取地点、推荐理由、避坑信息的效果还不够稳定，提取结果可能存在不完整、不准确或不够自然的问题。

问题分析：

当前 V2 版本默认使用前端 Mock 数据和简单演示逻辑，攻略文本提取功能主要用于展示流程，并没有接入真实 AI API，也没有对复杂攻略文本进行充分解析。因此在面对较长文本、口语化内容、多个地点混杂、图片 OCR 内容或平台笔记风格文本时，提取效果可能不够理想。

V2.1 处理：

- 优化粘贴攻略面板的说明文案，明确当前是演示提取。
- 改善前端 Mock 提取逻辑，按换行、顿号、逗号、序号等进行简单拆分。
- 对短文本增加友好提示。
- 真实 AI 解析能力保留到 V4。

---

### 2. 用户希望知道推荐数据来源，并希望后续关联旅游攻略类网站

反馈内容：

用户反馈“这个推荐是哪里的数据哇，可以关联一下那种导游网站可能”。说明用户会关心推荐地点的来源，也希望系统能结合真实旅游攻略、导游网站、地图平台或用户分享内容进行推荐。

问题分析：

当前 V2 版本使用的是前端内置 Mock 数据，推荐地点主要用于演示完整流程，并不是来自真实 AI、高德地图、小红书、马蜂窝、携程、去哪儿、Tripadvisor 或其他旅游攻略平台。虽然页面中有“Mock 数据”标签，但仍需要更明确地说明当前数据来源。

V2.1 处理：

- 在候选地点推荐区域增加数据来源说明。
- 文案明确写明当前为 V2.1 内置 Mock 演示数据。
- 后续真实 AI、地图和攻略数据源接入保留到 V4。
- 对攻略网站和平台内容，后续优先采用用户手动粘贴或授权方式，不做违反平台规则的自动爬取。

---

### 3. 地点页面内容较长，手机端浏览有点累

反馈内容：

用户反馈“很多地点页面很长，翻的会有点烦，可以做成小点的标签之类的形式”。说明当前候选地点卡片信息量偏大，手机端浏览多个地点时需要频繁上下滑动。

问题分析：

V2 卡片展示了地点名、类型、推荐理由、场景、停留时间、地址、避坑提示和操作按钮，信息完整但在手机屏幕上较长。

V2.1 处理：

- 将类型、来源、停留时间、适合场景改为标签形式。
- 默认只展示地点名、标签、简短推荐理由和操作按钮。
- 地址、避坑提醒收进“详情 / 收起”。
- 优化手机端按钮、卡片间距和文字换行。

---

### 4. 用户希望结合天气预报做出行提醒

反馈内容：

用户希望系统能结合天气预报，如果目的地可能下雨，可以提醒谨慎安排某些景点。

问题分析：

当前 V2/V2.1 没有接入天气 API，也不会根据天气调整行程。对于自然风景区、山地景区、户外步行路线、观景台、海边、古镇、夜市等场景，天气确实会明显影响体验。

V2.1 处理：

- 暂不接入天气 API。
- 在 README 和 CHANGE_RECORD 中记录该需求。
- 后续 V4/V5 可加入天气预报、雨天备选和户外风险提醒。

---

### 5. 预算只用低 / 中 / 高表达有点笼统

反馈内容：

用户反馈“预算的话低中高可能有点笼统，不知道能不能量化”。说明用户希望预算设置更具体，而不只是三个模糊档位。

问题分析：

当前版本预算档位主要用于快速完成问答流程。真实旅行规划中，预算通常需要更明确，例如人均总预算、每日预算、住宿预算、餐饮预算、交通预算等。

V2.1 处理：

- 保留低 / 中 / 高选项。
- 在预算问题附近增加解释：
  - 低预算：尽量选择公共交通、平价餐饮、免费或低门票景点。
  - 中预算：兼顾体验和性价比。
  - 高预算：优先体验质量，可接受更高交通、餐饮和门票成本。
- 预算量化输入保留到后续版本。

### 6. 推荐地点存在重复和过细 POI 问题（V4.3.2 处理）

反馈内容：

用户在测试昆明目的地时反馈：推荐结果出现「云南大学东门 / 西门 / 北门 / 停车场」「翠湖公园南门 / 停车场」「金马坊停车场」等条目，看起来像多个不同的景点；用户真正想看的是「翠湖公园」「金马坊」「云南大学」这类主地点。

问题分析：

景区入口、停车场、出租车上客点、公交站、卫生间等附属 POI 不应该作为候选景点单独列出。它们本质上是同一个主地点的附属设施，会让推荐列表显得重复、像 POI 数据库导出，不像真实旅行规划结果。

V4.3.2 处理：

- 新增地点名称归一化、低价值 POI 判定、候选地点去重与主地点聚合工具。
- 前端 Mock 推荐 / 后端推荐 / 真实高德 POI 搜索三条链路统一调用去重逻辑。
- 高德 POI 收集入口处直接跳过明显附属 POI（停车场 / 公交站 / 出入口…），同时节约后续 AI 文案增强的 token。
- 行程生成前也做一次轻量去重，避免同一主地点被分到同一天的多个时段。
- 新增昆明城市专属 Mock 数据，避免出现一堆门和停车场。
- 候选地点页加一行温和说明，告诉用户已经做了基础 POI 去重。

不会做的：

- 不做坐标聚类（真实坐标聚合留到接入真实地图后）。
- 不强行删除「中华门 / 朝阳门 / 西直门 / 宣武门 / 玄武门 / 凯旋门 / 前门大街 / 东门老街」这种本身就是知名主地点的名字。

## V2.1 处理范围

V2.1 只做轻量优化，不引入真实后端和真实 API。

本轮包含：

- 优化 Mock 推荐数据，减少明显模板拼接感。
- 增加部分热门城市的城市专属 Mock 地点。
- 增加更明确的数据来源说明。
- 优化候选地点卡片展示方式，减少手机端滑动负担。
- 将部分地点信息改成更紧凑的标签或详情形式。
- 优化手机端按钮、卡片间距和文字层级。
- 记录天气预报需求。
- 记录预算量化需求。
- 更新 README、CHANGE_RECORD 和 AGENTS。

- 自动根据天气调整行程。
- 预算精细化计算。

- PWA 安装。

## 后续版本方向

- V3：部署 FastAPI 后端，为真实 AI、地图 API 和服务端能力做准备。
- V4：接入真实 AI API 和高德地图 API，优化地点推荐、地点定位、路线规划和攻略文本解析。
- V5：接入天气预报能力，增加雨天备选、户外风险提醒和预算量化输入。
- V6：如有真实使用需求，再考虑 PWA 或 Capacitor 打包。
